uv: The Python Package Manager Worth Switching To

March 6, 2026 · 6 min read · python tooling

If you've ever waited 30 seconds for pip install to resolve dependencies, you'll appreciate uv. It's a Python package manager written in Rust by the team behind Ruff, and it's genuinely fast — often 10–100x faster than pip.

I started using it while setting up my dev machine and haven't looked back. This post covers installation, everyday workflows, and where it fits alongside existing tools.

Installation

The official installer is a single curl command:

curl -LsSf https://astral.sh/uv/install.sh | sh

After that, restart your shell (or run source ~/.bashrc) and confirm it's available: 

uv --version
# uv 0.5.x

Creating a virtual environment

uv replaces both pip and venv. To start a new project: 

# Create a venv in .venv/
uv venv

# Activate it (same as always)
source .venv/bin/activate

You can also target a specific Python version even if it's not installed yet — uv will download it automatically: 

uv venv --python 3.12

Installing packages

Drop-in replacement for pip install: 

# Install a single package
uv pip install requests

# Install from requirements.txt
uv pip install -r requirements.txt

# Install with extras
uv pip install "fastapi[standard]"

The first run fetches packages; subsequent runs hit a local cache so repeated installs are nearly instant.

Running scripts without a venv

One of my favourite features is uv run, which creates a temporary isolated environment just for that script: 

# Run a script, auto-installing its deps
uv run script.py

# Specify dependencies inline
uv run --with requests --with rich script.py

This is handy for one-off scripts you don't want polluting a shared environment.

Managing a project with uv

uv also understands pyproject.toml and can replace the full pip + venv + pip-tools workflow: 

# Initialise a new project
uv init my-project
cd my-project

# Add a dependency (updates pyproject.toml + uv.lock)
uv add httpx

# Remove a dependency
uv remove httpx

# Sync the environment to the lockfile
uv sync

The lockfile (uv.lock) is cross-platform and deterministic — commit it to version control so teammates get identical environments.

Speed comparison

Installing a typical data-science stack to demonstrate the difference:

# pip (cold cache)
time pip install numpy pandas scikit-learn matplotlib
# real: 1m 14s

# uv (cold cache)
time uv pip install numpy pandas scikit-learn matplotlib
# real: 4.2s

# uv (warm cache)
time uv pip install numpy pandas scikit-learn matplotlib
# real: 0.3s

The warm-cache case is the daily reality — near-instant reinstalls.

uv vs pip vs pipenv vs Poetry

  • pip — still needed for edge cases; uv is mostly a drop-in
  • pipenv — solves the same problem, but slower and less maintained
  • Poetry — great for publishing packages; uv is better for app workflows
  • uv — fastest, lowest friction, good enough for most projects

When NOT to use uv

  • Publishing packages to PyPI — Poetry or Flit still have better build-system integration
  • Complex Conda environments with non-Python binaries — stick with Conda/Mamba
  • Anywhere the team is fully committed to Poetry — consistency matters more than speed

Summary

uv is the tool I recommend for any new Python project. It's fast, it's compatible with the existing ecosystem, and it removes most of the friction that makes Python environment management frustrating. Install it once and let it handle the rest. 

# TL;DR cheatsheet
curl -LsSf https://astral.sh/uv/install.sh | sh  # install
uv venv                                            # create env
uv pip install <package>                          # install package
uv run script.py                                   # run without activating
uv init / uv add / uv sync                        # full project workflow
